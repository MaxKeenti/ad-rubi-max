04 - Modelo de datos

Firestore y Storage

El modelo conserva solo lo necesario para una observacion de bache: evidencia visual, coordenadas, severidad opcional, descripcion opcional, identidad anonima y metadatos de confirmacion/borrado.

Coleccion principal

reportes/{reporteId}
  lat, lng: Double
  geohash: String
  accuracyMeters: Double
  severidad: String?        # leve | moderado | severo
  descripcion: String?      # max 200 caracteres
  fotoPath: String          # reportes/{reporteId}.jpg
  fotoUrl: String
  createdBy: String         # Firebase anonymous uid
  confirmCount: Long
  serverWrittenAt: Timestamp
  deletedAt: Timestamp?
  deletedBy: String?

Subcoleccion de confirmaciones

reportes/{reporteId}/confirmaciones/{uid}
  confirmedAt: Timestamp

El documento usa el uid como ID. Esto impide confirmar dos veces desde la misma instalacion y evita una consulta adicional solo para saber si el usuario ya confirmo.

Storage

Cada foto se guarda como reportes/{reporteId}.jpg. La app comprime a JPEG con lado largo maximo de 1600 px antes de subir. El orden de escritura es intencional: primero Storage, despues Firestore. Asi no se crea un reporte que apunte a una foto inexistente.

Reglas principales

RecursoReglaMotivo
reportes createSolo firmado y createdBy == auth.uid, con forma valida.Evita crear reportes a nombre de otro uid.
reportes updateSolo borrar logicamente por autor dentro de 24 h, o incrementar confirmCount exactamente en 1.Reportes inmutables salvo operaciones permitidas.
reportes deleteDenegado al cliente.No hay borrado fisico desde app.
confirmacionesCreate si el doc ID es auth.uid; update/delete denegados.Una confirmacion es historica.
StorageWrite firmado, JPEG y tamano controlado; lectura firmada.Protege costo y evita archivos arbitrarios.
Brecha residual aceptada. Las reglas permiten incrementar confirmCount sin validar con getAfter() que exista la confirmacion del mismo batch. Se documento como riesgo menor porque el impacto seria inflar un contador, no tomar control de datos sensibles.
