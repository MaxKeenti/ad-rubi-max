03 - Arquitectura

MVVM Compose con contratos testeables

BacheWatch usa una arquitectura MVVM con Compose. Las pantallas observan estados de ViewModel y los ViewModels dependen de interfaces de repositorio, ubicacion y sesion. Hilt decide si se usan fakes o implementaciones reales, lo que permite pruebas unitarias sin Firebase y una app real con servicios remotos.

Contratos principales

ContratoResponsabilidadImplementaciones
ReporteRepositoryCrear reporte, observar viewport, recientes, confirmar y eliminar.Fake en memoria y Firestore/Storage real.
LocationProviderObtener una ubicacion actual precisa para reportar.Fake y Fused Location Provider.
SesionAnonimaAsegurar uid anonimo y exponer identidad para reglas.Fake y Firebase Auth anonimo.

Flujo de reporte

  1. El usuario abre Reportar.
  2. La app toma una foto con intent de camara del sistema.
  3. Se solicita ubicacion precisa con Fused Location Provider.
  4. La foto se comprime a JPEG de maximo 1600 px.
  5. Se sube la foto a Storage.
  6. Si la subida fue exitosa, se escribe el documento en Firestore con geohash y timestamp de servidor.
  7. El mapa y recientes actualizan por observacion de repositorio.

Por que existen fakes si Max desarrolla todo

Las interfaces no se conservaron por paralelismo, sino por testabilidad. Los fakes permiten probar ViewModels, estados de UI y filtros sin depender de red, Firebase, GPS o camara. Tambien permiten una demo deterministica con datos sembrados de CDMX.

Escalabilidad del mapa

El mapa no descarga todos los reportes. Cada reporte guarda un geohash; al mover el viewport se calculan rangos geohash, se ejecutan consultas por rango y se filtran falsos positivos en el cliente. Asi el costo de lectura escala con la zona visible, no con el tamano total de la base.
